Executive Summary

  • Found and used a public multi-million scene: bicycle from NerfBaselines/Mip-NeRF360, with 5,998,962 splats. Source: Hugging Face NerfBaselines Gaussian Splatting checkpoints.
  • Replaced the binary PLY hot path with direct field extraction. This avoids per-row dictionaries and made 6M-splat loading practical in the app and CLI.
  • Added budgeted streaming and budgeted sampled GPU-sort modes. Full 6M rendering is too slow at 156.01 ms/frame; 500k streaming reaches 15.72 ms/frame.
  • Added coarse shader culling and kept it enabled. A 500k culling-on run measured 12.83 ms versus 16.73 ms with culling disabled in the A/B check.
  • Steady state: 500k streaming + coarse culling is the app default for scenes above 1M splats. 250k is faster (10.39 ms) but visibly sparser; GPU-sort improves ordering but costs 18.93 ms at 500k.

What Changed

  • Milestone 7 optimization loop: added warmup-aware benchmark summaries, load/renderer-load timings, large-scene benchmark runs, lazy order-buffer initialization, direct binary PLY extraction, and coarse shader culling.
  • Milestone 8 experiments: added explicit streaming mode, visible-splat budgets, sampled GPU sort over only the budgeted subset, and coarse clip-space culling. Full tile/bin compositing was not promoted because the current bottleneck after budgeting is visual-density tradeoff and fragment cost, while tile binning would introduce blending-order risk and a much larger rewrite.
  • Viewer UX: large scenes load on a background task, auto-switch to streaming mode, set a 500k interactive budget, and show a persistent status message instead of freezing or silently changing modes.

Steady-State Decision

The renderer has reached a pragmatic steady state for this implementation generation. Further gains are possible only by accepting visible sparsity below 500k splats, dropping alpha-order quality with unsorted rendering, or replacing the renderer with a true tile/bin rasterizer and order-independent/transmittance compositing path. Those are larger algorithmic changes with correctness risk, so they should be treated as the next architecture phase rather than more tuning of this pass.
