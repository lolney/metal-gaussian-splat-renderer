GPU Radix Render Path Report

The default gpu mode is now a real GPU radix pipeline. The old global bitonic path remains available as bitonic for reference and QA. Raw JSON/CSV results are saved under Reports/gpu-radix-bench.

Executive Decision

GPU radix is implemented and kept as the default GPU sort path. It is competitive with bitonic at the current 100k-500k benchmark budgets, with small wins on the smaller cases and near parity/slight regression on the 500k bicycle case. This is the right architecture to continue optimizing because it removes the bitonic O(n log^2 n) scaling shape, but the current implementation is still a first production pass rather than a fully tuned radix sorter.

Implementation

Depth Keys

Depth-key generation still runs on GPU and writes SortPair(key,index) into the per-frame order buffer.

Radix Passes

Four 8-bit LSD passes sort 32-bit keys descending. Each pass uses histogram, prefix, bucket-start, and stable scatter dispatches.

Stable Scatter

Scatter computes a per-thread local rank inside a 256-item block, then combines bucket start, block offset, and local rank.

Reference Path

bitonic remains callable from the app and CLI, so QA and benchmarks can compare old and new global GPU sorters.

Refinement Pass: Prefix Bottleneck

The first GPU radix smoke used a serial GPU prefix kernel over all bucket/block histograms. The refinement parallelized that work by assigning one thread per bucket, then running a tiny bucket-start kernel.
