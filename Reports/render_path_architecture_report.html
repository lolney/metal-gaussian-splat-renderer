Render Path Architecture Report

Benchmarked on Apple M2 Pro with raw JSON/CSV saved under Reports/render-path-bench. The final default path is the phase-split renderer; radix, tiled, and projection-cache are explicit experimental paths.

Executive Decision

No experimental path is promoted as the default renderer. The accepted change is the explicit phase split plus new opt-in runtime paths that make the next GPU work measurable without contaminating the stable GPU bitonic and unsorted paths. Projection/covariance improves the representation but currently costs too much. CPU tiled/radix prototypes prove APIs and tradeoffs, but need GPU implementations before they can replace bitonic at large active counts.

Point-by-Point Decisions

1. Bitonic vs radix

Added radix mode using four 8-bit passes over 32-bit depth keys. It proves the runtime slot and data path, but the CPU prototype is not competitive with GPU bitonic. The right next step is moving histogram, prefix sum, and scatter to Metal.

2. Middle ordering

Added tiled mode that bins sampled splats into 32 px screen tiles and 16 depth buckets. It is a measurable approximation path but should stay experimental until binning is GPU-side.

3. Six vertex reads

Added optional useProjectionCache / --projection-cache. It projects each ordered splat once into a compact screen-space buffer and renders from that representation.

4. Covariance quality

The projection cache computes a projected 2D covariance from all three rotated scale axes, solves the 2x2 eigenbasis, clamps radius, and stores ellipse axes for rasterization.

5. Render phases

The accepted default code now has explicit frame planning, ordering/binning, projection/culling, raster/composite, and stats phases. Scene upload was already separated in SceneResources.
