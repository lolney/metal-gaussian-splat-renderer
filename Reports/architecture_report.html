Metal Gaussian Splat Renderer Architecture

This report describes the current architecture of the SwiftPM macOS Gaussian splat renderer: scene loading, static and per-frame Metal resources, sort modes, shader execution, in-app profiling, and the headless benchmark path.

Repo: metal-gaussian-splat-renderer Commit: abc343e Targets: SwiftUI app, renderer library, splatbench

Executive Summary

Renderer Core

SplatRenderer owns Metal device state, render and compute pipelines, static scene GPU resources, and a ring of mutable per-frame resources.

System Map

The app and benchmark both drive the same renderer API. The app embeds MTKView in SwiftUI and records live frame stats; the benchmark creates a renderer, loads the same scenes, renders offscreen frames, and writes timing files. The shared draw path is anchored at SplatRenderer.draw.

Frame Execution Pipeline

The ring matters because the order buffer is both written and read during frame execution. The renderer waits for an available in-flight slot, attaches a command-buffer completion handler, and only then reuses the corresponding FrameResources. Resource Ownership

Static Scene Resources

The renderer's long-lived GPU scene state is intentionally narrow: a private splat buffer plus counts. Upload happens through a shared staging buffer and a blit command, after which the packed CPU array can fall out of scope.

  • makeSceneResources creates the private splat buffer and staging upload.
  • SceneResources stores only the Metal buffer and scene size metadata.
  • load(scene:) waits for in-flight frames before replacing scene resources.

Per-Frame Mutable Resources

Mutable resources live in FrameResources: each frame slot owns one uniform buffer and per-mode order-buffer state. Switching sort modes clears that frame slot's mode state before allocating a new order buffer.

  • initial uniform ring allocation
  • order buffer allocation and mode reset
  • FrameResources fields
Architectural invariant:

SceneResources are replaced only after all in-flight frames drain. FrameResources are reused only after the GPU signals the corresponding command buffer. That separates static GPU-read-mostly state from per-frame mutable state.

Sort Modes

Mode Where order comes from Tradeoff
gpu depth-key encoder followed by bitonic sort dispatches, running kernels defined at ShaderSource lines 63-115. Best runtime path for sorted output at interactive budgets; sort work scales with padded active draw count.
cpu SplatScene.sortedIndices computes far-to-near order, then updateCPUOrder writes the order buffer. Useful as a correctness/reference path; intentionally available even when slow on large scenes.
unsorted updateUnsortedOrderIfNeeded and writeUnsortedOrder write sampled scene indices and sentinel-fill unused entries. Avoids depth sorting entirely; useful for very large scenes, profiling draw cost, and quick inspection.

Shader Architecture

Compute

The depth-key kernel maps active instance IDs to source splat indices, transforms splat centers into view space, and writes a SortPair. The bitonic kernel then performs in-place pairwise swaps.

  • depthKeyKernel
  • bitonicSortKernel
  • compute dispatch sizing

Rasterization

The vertex shader emits a camera-facing quad per splat instance, uses the order buffer to select the splat, projects radius into pixels, applies optional coarse culling, and clamps splat size. The fragment shader evaluates the Gaussian falloff and pre-multiplied color.

Architecture Notes and Open Edges

Correctness Foundation

Per-frame order buffers remove the inter-frame race class for GPU, CPU, and unsorted modes. Mode switches are intentionally treated as resource-state boundaries.

Performance Shape

GPU sorting is still bitonic and globally ordered, which is simple and measurable but not the likely final large-scene architecture. The current design makes tile/bin or hierarchical experiments easier to add behind runtime flags.

Memory Shape

Static splats are now GPU-private after upload. CPU sort keeps source splats, which is deliberate while CPU mode remains user-selectable after load.
