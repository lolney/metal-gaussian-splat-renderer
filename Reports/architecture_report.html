Metal Gaussian Splat Renderer Architecture

Current-state architecture report for the SwiftPM macOS Gaussian splat renderer after the GPU radix refactor and cleanup pass. The active product shape is a reusable Metal renderer, a SwiftUI viewer, and a headless benchmark runner with profiling data available in both JSON/CSV and the app UI.

Repo: metal-gaussian-splat-renderer Source snapshot: 96b5f09 Report updated: May 18, 2026 Primary path: GPU RADIX

Executive Summary

The renderer now has a tighter runtime surface: unsorted, radix, gpu, and bitonic. The prior tiled experiment and non-radix CPU sort mode have been removed from the product/UI/CLI surface. CPU radix remains as a reference path; GPU radix is the default path.

Resource ownership is explicit enough for continued optimization: static scene splats upload to a private Metal buffer, per-frame mutable state lives in a three-frame resource ring, mode switches reset order state, and headless benchmarks exercise the same renderer code used by the app.

Frame Pipeline

Scene Upload

Pack Splat values, stage through shared memory, blit into a private splatBuffer.

Frame Plan

Clamp options, choose draw count, compute padded count only when the chosen mode needs it.

Ordering

GPU radix, bitonic reference, CPU radix reference, or sampled unsorted order.

Projection

Optional compute pass projects each selected splat into compact screen-space data.

Raster + Stats

Composite instanced quads, commit, wait when profiling, and return timing/memory estimates.

Resource Ownership

Scene Lifetime

  • SceneResources owns the private Metal splat buffer and scene count.
  • cpuSortSplats is retained only to support CPU radix reference mode.
  • Loading a new scene waits for the resource ring to drain before replacing scene resources.
  • Offscreen render targets are cached by size and kept in private GPU storage.

Frame Lifetime

  • Three FrameResources rotate behind an in-flight semaphore.
  • Uniforms are shared per-frame buffers.
  • Order buffers are mode-local and reinitialized when sort modes change.
  • GPU radix scratch, histograms, offsets, bucket totals, and bucket starts are per-frame resources.
  • Projection-cache output is a private per-frame buffer sized to active draw count.

Sort Modes

ModeStatusImplementationUse case
gpu / GPU RADIX DEFAULT GPU depth-key generation plus four 8-bit radix passes over SortPair. Primary interactive and benchmark path for sorted rendering.
radix / CPU RADIX REFERENCE CPU-generated depth keys and CPU radix ordering, written into the per-frame order buffer. Correctness/reference path for small scenes and debugging. Not expected to be fast at 100k+ splats.
bitonic / BITONIC REFERENCE GPU depth-key generation plus legacy global bitonic sort. Comparator for GPU radix behavior and performance. Still useful because it stresses command and memory traffic differently.
unsorted / UNSORTED PREVIEW Sampled source order only; no depth-key or sort work. Large-scene preview, profiling isolation, and visual sanity checks.
tiled and old cpu REMOVED Removed from enum, app picker, CLI help, benchmark script, and tests. Future tile/bin work should return as a proper GPU architecture, not a productized CPU-side approximation.

GPU Radix Path

Current Design

  • depthKeyKernel writes (depthKey, splatIndex) pairs.
  • Each radix pass clears histograms, builds per-block histograms, computes offsets, computes bucket starts, and scatters.
  • After four 8-bit passes, the sorted result lands back in the frame order buffer.
  • Scratch and histogram resources are allocated per frame resource and grow to capacity.

Known Limits

  • The prefix and bucket-start stages are intentionally simple, not a fully optimized parallel scan.
  • The path remains global-order based; it has not yet become a tile-local compositor.
  • Sorting still happens every frame for gpu, even for camera changes where approximate ordering might be enough.
  • Memory-bandwidth estimates are model estimates, not hardware counter reads.

Projection And Raster

Baseline Vertex Path

Draws six vertices per splat instance. The vertex shader reads the sorted pair and packed splat, computes covariance ellipse axes, and emits a camera-facing quad.

Projection Cache

Optional compute prepass projects selected splats once into ProjectedSplat. The projected render pipeline then reads compact screen-space data.

Quality Model

The shader now uses a screen-space covariance/eigenvalue treatment for ellipse axes. Radius clamp and coarse frustum culling remain runtime options.

Profiling Surface

In-App Panel

Shows frame history, GPU summary, memory estimate, CPU encode-stage bars, scene stats, and controls for pause, clear, export, profiling, wait-for-GPU, projection cache, budget, and max radius.

CLI Output

splatbench supports --sort unsorted|radix|gpu|bitonic, --projection-cache, --max-splats, JSON/CSV output, and optional .gputrace capture.

Measured Fields

Frame stats include total frame, CPU encode, GPU command buffer, depth-key, sort, projection, draw, visible/total splats, estimated bytes, and estimated GB/s.

Remaining Architectural Risks

  • GPU radix is a better long-term fit than bitonic, but the current implementation is still a straightforward first version, not a tuned production sort.
  • The renderer still uses global ordering. A future tile/bin renderer should be introduced as a clean GPU phase after the current radix path is stable.
  • CPU radix keeps source splats resident. That is acceptable for a reference path, but very large scenes still pay CPU memory overhead.
  • Projection cache is available behind a flag; it should stay optional until benchmarks show clear wins across representative scenes.
  • Memory bandwidth visualization is useful for direction, but hardware-counter profiling in Instruments/Xcode should remain the source of truth for bandwidth bottlenecks.
